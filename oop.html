class User:

Unit 3: UML

Discuss which UML models are most applicable at different stages of the Software Development Life Cycle.

UML models are tools that help visualise and plan software systems at different stages of development. The choice of which models to use depends on the specific project and the needs of the developers and testers involved. However, the general use of UML models during the Software Development Life Cycle could be:

  1. Requirement analysis:

    2. Use Case Diagrams: These diagrams help identify and describe what the software needs to do from the users' perspective.

  2. Planning:

    3. Class Diagrams: These diagrams show the different parts of the system, like classes and their relationships, helping us plan how the system will work.
    Sequence Diagrams: These diagrams illustrate the steps and interactions between different parts of the system, helping us understand the order of actions.
    Activity Diagrams: These diagrams show the flow of activities or steps in a process, helping us visualise how the system should work.

  3. Software design:

    4. Class Diagrams: These diagrams provide more detailed information about the structure of the system, helping developers write the actual code.
    Sequence Diagrams: These diagrams help developers understand how different parts of the system should interact with each other, which is important for coding.

  4. Software development:

    5. Class Diagrams: Developers use these diagrams to guide them when writing the code for the system.

  5. Testing:

    6. Use Case Diagrams: These diagrams are useful for creating test cases to make sure the software works as intended.
    Sequence Diagrams: Testers can use these diagrams to design tests based on the expected interactions between different parts of the system.

  6. Deployment:

    7. Deployment Diagrams: These diagrams show how the software components are deployed on hardware, helping with the setup and configuration of the system.

Lange, C. F. J., Chaudron, M. R. V. & Muskens, J. (2006) In practice: UML software architecture and design description. IEEE Software 23(2): 40-46.

Unit 6: Abstract Methods and Interfaces

Discuss the metrics used to assess the features of an object oriented program.

When assessing the features of an object-oriented program, several metrics are commonly employed to evaluate its quality, complexity, maintainability, and other aspects. These help people understand the codebase and make informed decisions for improvement. (Shaheen et al., 2019). The following metrics are commonly employed to assess different aspects of the program:

  • Size: Assessing the size such as population (number of classes and operations), volume (dynamic object count), length (e.g., depth of inheritance), and functionality (number of user functions), provide insights into the scale and scope of the program. Smaller classes are easier to comprehend and maintain (Henderson-Sellers, 1995).
  • Cyclomatic Complexity: This metric calculates the number of independent paths through the code, helping identify complex areas that may be error-prone and challenging to understand. Lower complexity indicates simpler code (Jamali, 2006 & Henderson-Sellers, 1995).
  • Coupling: This quantifies the interdependence and interactions between classes. Coupling metrics provide insights into the level of collaboration and communication between classes (Jamali, 2006 & Henderson-Sellers, 1995).
  • Cohesion: Cohesion metrics focus on how well a class's attributes and operations align to achieve a single, well-defined purpose within the problem domain. High cohesion implies that a class is focused, organized, and encapsulates related responsibilities effectively (Jamali, 2006).
  • Volatility: Volatility metrics assess the likelihood of changes occurring in the design or implementation of a class. Classes with high volatility require frequent modifications, highlighting areas that may require additional attention and maintenance (Jamali, 2006).
  • Sufficiency: It is used to assess the features of an object-oriented program and evaluates whether a class has the necessary properties to solve the problem it is required to solve. It aims to determine if the class adequately fulfils its intended purpose and requirements (Jamali, 2006).
  • Primitiveness: Primitiveness or simplicity metrics measure the extent to which class operations cannot be composed from other operations. A class with high primitiveness has simpler and more independent functionality (Jamali, 2006).
  • Completeness: Evaluates whether a class reflects all the properties and aspects of the problem domain, with the intention of enabling reuse (Jamali, 2006).
  • Similarity: Compares the structure, function, and behaviour of two or more classes. It helps identify similarities or differences between classes, helping with code organization, design decisions, and potential reuse (Jamali, 2006).

By using these metrics, we can gain insights into the strengths and weaknesses of object-oriented programs, identify areas for improvement, and make informed decisions to enhance the quality and maintainability.

References:

Henderson-Sellers, B., 1995. Object-oriented metrics: measures of complexity. Prentice-Hall, Inc..

Jamali, S.M., 2006. Object oriented metrics. A survey approach Technical report, Department of Computer Engineering, Sharif University of Technology, Tehran, Iran.

Shaheen, A., Qamar, U., Nazir, A., Bibi, R., Ansar, M. and Zafar, I., 2020. Oocqm: object oriented code quality meter. Computational Science/Intelligence and Applied Informatics, pp.149-163.

from abc import ABC, abstractmethod

class Bank(ABC):
    @abstractmethod
    def deposit(self, amount):
        pass
                                        
    @abstractmethod
    def withdraw(self, amount):
        pass
                                        
    @abstractmethod
    def check_balance(self):
        pass
                                    
class SavingsAccount(Bank):
    def __init__(self):
        self.balance = 0
                                        
    def deposit(self, amount):
        if amount > 0:
            self.balance += amount
            print(f"Deposited {amount} into the savings account.")
        else:
            print("Invalid amount. Please enter a positive value.")
                                        
    def withdraw(self, amount):
        if amount > 0:
            if self.balance >= amount:
                self.balance -= amount
                print(f"Withdrew {amount} from the savings account.")
            else:
                print("Insufficient funds.")
        else:
            print("Invalid amount. Please enter a positive value.")
                                        
    def check_balance(self):
        print(f"Current balance in the savings account: {self.balance}")
                                    
account = SavingsAccount()
                                    
account.deposit(100)
account.check_balance()
account.withdraw(50)
account.check_balance()
account.withdraw(100)
account.check_balance()
account.deposit(-10)

def handle_errors():
    try:
        # do something
        number = int(input("Enter a number: "))
        result = 10 / number
        print("Result:", result)
                                
    except ValueError:
        # handle ValueError exception
        print("Invalid input. Please enter a valid number.")
                                
    except (TypeError, ZeroDivisionError):
        # handle multiple exceptions (TypeError and ZeroDivisionError)
        print("An error occurred during the calculation.")
                                
    except Exception as e:
        # handle all other exceptions
        print("An exception occurred:", str(e))
                                
def main():
    # Call the method that handles errors
    handle_errors()

Unit 12: Working with Design Patterns to Structure Code

Read Zhang & Budgen (2012). Which design patterns are used most commonly, and why?

The design patterns that are most commonly used in software design are Composite, Observer, and Visitor. These patterns are popular because they are versatile and therefore are used in many software development projects. They have a significant impact on how software systems are structured and designed when implemented.

One reason why The Gang of Four (GoF) patterns, which includes (Composite, Observer, and Visitor) are the most commonly used is because they come with detailed guidelines and examples. The GoF patterns are also widely used because they offer well-established and proven solutions to common design problems that software developers can often encounter. By using design patterns, they can improve code reusability, maintainability, and scalability which means more flexible software systems.

References:

Zhang, C. & Budgen, D. (2012) What Do We Know about the Effectiveness of Software Design Patterns? in IEEE Transactions on Software Engineering 38(5): 1213-1231. DOI: 10.1109/TSE.2011.79.
